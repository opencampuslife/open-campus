## Architecture
```
Admin Console (Refine + shadcn + React Flow)
    │  HTTP /api/admin/*
    ▼
BFF (server.py + admin_gateway.py)  ←── admin role check
    │
    ├── Source Ingestion Service
    │   ├── upload → docling → frontmatter check → compliance → staging
    │   └── approve → publish → knowledge_vault (+ PostgreSQL)
    │
    ├── Knowledge Graph Service
    │   └── scan vault → build graph → store nodes/edges
    │
    ├── MCP Server (ua-mcp-server, stdio)
    │   └── scan_source / build_graph / review_graph (sandboxed)
    │
    └── Audit Log (JSONL)
```

## Security Boundaries

| Boundary | Design |
|---|---|
| BFF is sole admin entry | All /api/admin/* routes gated by admin role |
| staging ≠ production RAG | staging docs are NOT in knowledge_index.json until published |
| MCP is sidecar only | MCP tools cannot write to knowledge_docs/chunks or staging frontmatter |
| RLS remains final boundary | Published docs still go through search_accessible_chunks() |
| Frontend role is untrusted | BFF uses trusted identity, not client-supplied role |
| MCP sandboxed paths | Only knowledge_vault/ and data/ingestion/ accessible |
| Publisher validated | Only approved + compliance-passed docs can be published |

## Known Limits

1. **Admin Console is MVP** — not production IAM; role model is admin-only, no content_operator/reviewer granularity
2. **PostgreSQL production index** — publisher writes to knowledge_vault + optionally to PostgreSQL; the chunk-then-embed-then-index loop uses the existing indexer flow
3. **Docling adapter** — has graceful fallback, but real PDF table quality needs sample validation
4. **graph_store uses JSON files** — production could migrate to PostgreSQL
5. **MCP server uses stdio** — multi-instance deployment would need HTTP wrapper
6. **URL validator** — format + reachability only, not full SSRF-grade
7. **Temporal / Tika** — intentionally not connected (correct MVP scope decision)
