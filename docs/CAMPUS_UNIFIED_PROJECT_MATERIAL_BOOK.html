一、项目定位与需求痛点

本项目面向学校日常高频事务处理场景，目标不是构建单一功能页面，而是搭建一个具备统一入口、统一状态流转、统一提醒催办、统一审计追踪和统一扩展能力的校园事务智能体平台。

在学校日常管理中，订退餐、请假、报修、招生咨询、材料收集等事务通常分散在微信群、Excel 表、电话沟通和人工转发链路之中。此类流程表面上看是“简单小事”，实际却具有参与角色多、附件材料多、流程节点多、时间要求强和责任边界模糊等特点，极易形成遗漏、重复确认、统计滞后和责任不可追溯的问题。

传统信息化系统往往以单模块、单表单、单审批链为核心，能够解决局部录入问题，但难以跨模块复用身份、权限、附件、提醒、审计和日报机制；而直接使用通用大模型进行问答式处理，又难以稳定完成真实业务场景中的状态流转、外部工具调用、超时提醒和结果落库。因此，学校需要的并不是单纯“会对话”的模型，而是具备任务理解、流程编排、规则执行、异常升级和结果汇总能力的智能体型应用系统。

表 1 项目核心痛点与对应解决思路
痛点类别 典型表现 本项目解决思路
入口割裂 家长、学生、教师、后勤、供应商分别通过不同工具沟通，信息难汇总。 通过企业微信、H5 和管理后台统一接入，建立同一身份与事务入口。

因此，本项目的总体定位可以概括为：以多智能体协同和统一业务底座为核心，将学校高频事务从零散沟通转变为标准化、自动化、可审计的数字化闭环。

二、作品设计与制作

2.1 总体架构

项目依托 gaokao-agent 已有的智能体运行底座，在招生问答智能体、权限隔离、知识检索和审计能力之上，扩展出面向校园事务的统一化业务层。其架构遵循“交互入口层—智能体编排层—业务服务层—基础设施层”的分层设计，使各类事务能够复用同一套鉴权、流程、提醒、审计与发布保障能力。

在方法论层面，本项目明确参考并继承 OpenHuman 的智能体产品与工程理念：以“记忆/知识—工具—流程编排”为核心，将能力沉淀为可复用的运行底座，再面向具体业务快速实现“可执行、可追溯、可审计”的闭环任务系统。基于这一理念，本项目不仅在形态上与 OpenHuman 同类（均为智能体驱动的任务型应用），也支持在后续阶段将部分能力按接口/模块维度迁移或对接到 OpenHuman 生态之中（例如知识治理、工具注册、评测门禁和审计规范等）。

2.2 智能体职责拆分

本项目并非将“智能体”作为营销概念，而是将其落实为可分工、可验证的流程角色。不同智能体围绕同一业务对象协同工作，保证各事务既能自动处理，又能在必要时被人工接管和审计。

表 3 智能体角色划分

入口识别智能体 企业微信身份、H5 请求、回调消息 识别学校、角色、用户绑定关系和入口类型 标准化身份上下文
流程推进智能体 结构化事务请求 根据不同模块状态机推进 submitted、approved、processing、closed 等状态 状态流转结果

审计安全智能体 关键操作、敏感字段、附件绑定行为 记录审计日志、校验越权风险、拒绝非法附件复用 审计事件与安全告警
汇总分析智能体 多模块业务数据和异常状态 生成日汇总、异常清单、模块健康状态 日报与看板数据

2.3 子母协同架构与子 Agent 机制

为适配校园事务“流程长、证据多、角色多、异常多”的特点，本项目采用子母协同（主 Agent + 子 Agent）机制：主 Agent 负责任务理解、计划生成与全局一致性；子 Agent 以“能力/视角/约束”为边界并行执行检索、规则校验、工具调用、审核与汇总等子任务。该机制既提高吞吐，也提升结果可控性与可追溯性。

User/WeCom/H5/Admin -> API Gateway (Auth/Scope/Idempotency) -> Orchestrator (Parent Agent) -> Plan (steps, tools, constraints) -> Spawn Sub-Agents (parallel) -> Retrieval Sub-Agent (KB/DB evidence) -> RuleCheck Sub-Agent (SLA/state/permission) -> ToolExec Sub-Agent (worker/OCR/RPA) -> Review Sub-Agent (conflict/format/safety) -> Summary Sub-Agent (merge + structured output) -> Finalize (decision + structured result) -> Persist (DB + operation_logs + reminders)
图 2 子母协同（主 Agent/子 Agent）总体架构与分工关系示意图
Parent Agent -> (A) Task Spec (JSON) -> (B) Evidence Pointers (doc_id/attachment_id/query plan) -> (C) Allowed Scope (school_id, role, permissions) -> (D) Output Contract (schema + required fields) -> Sub-Agent -> Execute (retrieve/check/tool) -> Return: - result_json - evidence_refs (allowed only) - risk_flags - audit_fields (action, biz_type, biz_id) -> Parent Agent -> Conflict Resolve (cross-sub-agent) -> Write operation_logs + reminders -> Respond (structured + explainable)

2.4 关键制作过程

项目制作遵循“底座先行、链路打通、再做扩展”的思路，而不是一次性堆砌全部业务模块。首先搭建统一数据库与权限策略，再完成 H5 提交、企业微信接入、提醒 Worker、附件入站、锁单确认、日报汇总等核心能力，最后通过发布门禁和试点流程将系统从 Demo 形态推进到可试运行形态。

2.6 面向新增模块的统一技术底座

根据新增方案，平台后续将进一步承接考试登分与成绩核对、材料收集与缺交催办、请假与销假闭环、缴费确认与对账辅助、考勤异常与晚自习点名等五类高频事务。这些模块虽然业务目标不同，但其底层处理范式高度一致，均可复用“采集—识别—匹配—规则校验—人工复核—提醒催办—审计留痕—导出汇总”的统一框架。

3.1 订退餐与供应商确认

订退餐模块是当前最完整的试点链路，覆盖家长/学生提交、后勤锁单、供应商确认、异常升级和导出汇总等步骤。该模块已经形成从前台提交到后台统计再到供应商闭环确认的完整链条，也是项目试点运行的首个验证场景。

订退餐端到端闭环 家长/学生提交订餐或退餐（H5/WeCom） -> 校验绑定关系与截止时间 -> 写 meal_orders + attachments（可选） -> 创建 reminder_tasks（确认/超时） -> 后勤锁单 meal_locks + 固化汇总快照 -> 生成 vendor_confirmation（签名链接） -> 供应商确认 confirmed/abnormal -> 异常升级 + 审计留痕 operation_logs -> 导出汇总 CSV / 日报统计

3.2 请假审批

请假模块支持家长或学生在线提交申请，班主任进行审批，教务留痕查询，并可扩展到销假提醒和异常统计。其价值在于减少班级群内反复确认和线下登记成本，使请假过程具有标准台账与责任记录。

请假审批与教务留痕 家长/学生提交请假（H5） -> 校验学生归属与时间范围 -> 写 leave_requests（pending）+ attachments（可选） -> 推送班主任审批（WeCom App Msg） -> 班主任 approve/reject -> 写 operation_logs（审批动作） -> 教务台账查询与导出 -> 进入日报汇总（按日/班级）
图 7 请假提交—审批—台账留痕 Pipeline（论文式线条图）

3.3 报修工单

报修模块将宿舍、教室和公共区域问题转化为标准工单，支持责任分配、状态更新和关闭确认。该模块尤其适合高频但易被口头沟通淹没的后勤场景，可显著提升事项追踪能力。

报修工单闭环 学生/教师提交报修（H5） -> 写 repair_tickets（pending_assign）+ attachments（可选） -> 通知后勤负责人（WeCom） -> 后勤 assign 维修人员（processing） -> 维修人员 complete（上传完成图，可选） -> 提交人/后勤 close 确认 -> 超时未分配/未完成 -> reminder_tasks 升级 -> operation_logs 全程留痕 + 看板统计
图 8 报修提交—分配—处理—关闭 Pipeline（论文式线条图）

3.4 管理看板与运维保障

系统支持管理后台列表页、健康检查接口、就绪检查接口、内部 worker 状态接口和统一审计日志。配合 `make release-check`、试点运行手册和阻断式 Benchmark，项目已具备从开发环境进入受控试点环境的基础条件。

每日日报汇总（跨模块） 定时触发（每天固定时间） -> 汇总 meals/leaves/repairs/materials/admissions 等数据 -> 识别未闭环与超时事项 -> 生成校级/年级/班级/后勤日报 -> 写 daily_reports -> 推送到对应负责人（WeCom） -> 管理后台可查询历史日报

3.6 考试登分与成绩核对

该模块面向教务和任课教师的成绩处理场景，重点解决 Excel 导分、截图抄分、纸质成绩单回录等过程中的错登、漏登、越界分数和学生匹配错误问题。系统将通过导入、识别、匹配、校验、复核和导出的完整链条，降低成绩处理的人为差错率。

教务创建考试任务 -> 导入学生名单和科目配置 -> 教师上传 Excel / 截图 / 成绩单照片 -> 解析成绩数据并匹配学生、科目 -> 校验分数范围、缺考标记、重复登分、姓名学号匹配 -> 正常项入库，异常项进入 review_items -> 教师逐项复核 -> 教务确认锁定成绩 -> 可选 RPA 回填旧教务系统 -> 导出成绩表与审计报告

3.7 材料收集与缺交催办

该模块用于报名表、承诺书、证件照、健康证明、竞赛材料和综合素质材料等任务型收集场景。系统将按班级和学生维度自动判断是否缺交、是否错交、是否重复上传，并通过 Reminder Worker 在截止前后持续催办和升级。

教师创建材料任务 -> 配置班级范围、材料类型与截止时间 -> 生成 H5 提交链接 -> 学生/家长上传材料 or 企业微信发图 -> OCR / 分类器识别姓名、学号、材料类型 -> 校验缺交、错传、命名不规范、重复提交 -> 正常材料归档，异常材料进入人工复核 -> 截止前提醒未交，逾期后生成缺交清单 -> 教师批量确认、退回或导出归档包

3.8 请假与销假闭环

现有请假能力将进一步从“提交与审批”延展到“请假到期提醒、销假确认、异常升级和教务统计”的完整闭环，使班主任、学生和教务都能围绕同一条状态链工作，减少群聊口头说明造成的遗漏。

请假 + 销假闭环 提交请假（pending） -> 班主任审批（approved/rejected） -> 若 approved：创建销假 reminder_tasks（end_time） -> 到期触发销假提醒（学生/班主任） -> 未销假超时：升级教务/年级组 -> 销假确认（returned/closed） -> 进入日报与统计 + operation_logs 留痕

3.9 缴费确认与对账辅助

该模块定位为缴费凭证识别与对账辅助工具，不承担实际支付功能，也不替代财务系统。它主要服务于班费、活动费、住宿费、校服费、资料费等非复杂支付场景，通过截图 OCR、金额匹配、流水导入和异常清单，帮助班主任和财务快速确认已缴、未缴和金额不符情况。

缴费确认与对账辅助（不含支付） 创建费用项（金额/对象范围/截止） -> 家长上传缴费截图（H5） -> OCR 提取金额/时间/交易号/备注姓名 -> 学生匹配（模糊匹配 + 规则） -> 导入财务流水（Excel，可选） -> 自动对账：已缴/未缴/异常金额/重复凭证 -> 人工复核确认（批量） -> 生成未缴名单 -> reminder_tasks 催办/升级 -> 导出对账表 + operation_logs 留痕

3.10 考勤异常与晚自习点名

考勤异常与晚自习点名 创建点名任务（session） -> 加载名单 + 同步请假数据 -> H5 快速点名 or 导入设备/Excel 数据 -> 规则引擎识别异常（迟到/缺勤/冲突） -> 生成异常台账 + 人工确认/修正 -> 触发通知（班主任/家长/年级组） -> 进入日报/月报统计 + operation_logs 留痕

四、项目创新点

表 7 项目主要创新点

统一事务底座上的多智能体协同 校园事务看似分散，实则重复依赖身份、提醒、附件、审计和状态流转能力。 项目将事务识别、流程推进、提醒升级、审计记录和汇总分析拆分为协同智能体角色，形成可复用的平台型架构，而非单点业务功能堆叠。

招生问答底座与校园事务底座融合扩展 招生咨询与校园事务通常由不同系统承接，数据与流程难复用。 项目依托 gaokao-agent 既有的问答、权限和 RAG 底座，为后续“咨询线索识别—跟进提醒—校内业务衔接”提供统一能力来源，兼顾对外咨询与对内事务管理。

上述创新并不局限于“使用了大模型”这一层面，而是体现在智能体方法如何与校园真实流程相耦合，如何与组织管理目标相匹配，以及如何以工程化方式进入可试点、可扩展、可审计的运行状态。
